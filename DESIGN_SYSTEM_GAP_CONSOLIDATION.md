# Krane Design System — Gap Consolidation

Consolidates `PROCUREMENT_LOG_GAP_ANALYSIS.md`, `SUBMITTAL_LOG_GAP_ANALYSIS.md`, and `DELIVERY_SCHEDULE_GAP_ANALYSIS.md` — read fresh in full for this document, not recalled from memory. Every finding below traces to one or more of those three reports; nothing new was investigated to produce this document, and no design-system code was touched to produce it.

**Two things change here that weren't true in the source reports.** First, duplicates are merged: several findings (the `DataTableRowActionsMenu` `aria-label` gap, `Button`'s missing `loading` prop, the `--spacing-*` token gap, and others) were independently reconfirmed across two or three reports — each appears here exactly once, with its full recurrence noted. Second, the ranking rubric is different. Each source report ranked findings by *that page's own* judgment of priority; this document re-ranks everything against one consistent rubric — **P0 = blocks product development, P1 = recurring workaround, P2 = quality improvement, P3 = enhancement** — which moves some items relative to where they sat in their original report (most notably: permission/role-gating and the calendar/timeline absence, never ranked #1 in any individual report, both rank P0 here, because the rubric's question — "does this block real product development," not "did this block *building a demo page*" — is a different, harder question than any single report asked).

---

## Components

Missing standalone primitives, found across all three pages.

- **No `Card` or stat-tile primitive.** *(P2 — quality improvement.)* Named in Procurement Log, reconfirmed unchanged in Submittal Log and Delivery Schedule. All three pages wanted a KPI/summary strip and built without one. `COMPONENT_STATUS.md` already records this as a deliberate deferral pending a concrete need — three pages independently wanting one is exactly the concrete need that discipline was waiting for.
- **No Drawer/Sheet (slide-in side panel).** *(P2.)* Named in Procurement Log, reconfirmed in both later reports. All three pages used a centered, blocking `Dialog` for "view detail" instead of a non-blocking side panel — a real, accepted compromise, not equivalent UX, recurring identically three times.
- **No permission/role-gating primitive anywhere in the system.** *(P0 — blocks product development.)* Named in Procurement Log, reconfirmed in both later reports. `Sidebar`, `AccountMenu`, and every `DataTable` row/bulk action render identically regardless of who's looking. This didn't block *building the three demo pages* — it blocks *shipping* any of the three real workflows they demonstrate (procurement approval, submittal review, delivery confirmation), all of which require hiding privileged actions from unauthorized users before they can go to production. Ranked P0 here specifically because the rubric asks about product development, not demo development.
- **`Badge` has exactly five non-default semantic variants**, a real ceiling. *(P2.)* Named once, in Procurement Log: `success`/`warning`/`destructive`/`info`/`pending`. All three pages' chosen status vocabularies (5, 3, and 5 statuses respectively) fit within it, but a sixth conceptually distinct status anywhere would force two meanings to share one color. Not yet hit; one constraint away.

## Tables

Everything specific to `DataTable` — by far the largest category, since all three pages were built on it.

- ~~A confirmed, reproducible sticky-column rendering defect.~~ **Resolved — see "Resolved" section below.**
- **Filter system has exactly three variants — `text`/`select`/`dateRange`.** *(P1 — recurring workaround.)* Named in Procurement Log, reconfirmed in both later reports. Every page's Vendor (or equivalent entity) filter used `select` as a stand-in for true searchable/type-ahead filtering, which only works because each page's demo dataset kept that column's options short enough to enumerate. A real entity list of any meaningful size breaks this workaround.
- **No numeric-range filter variant.** *(P3 — enhancement.)* Named once, in Procurement Log ("Amount between $X and $Y"). No page built so far has needed it badly enough to work around — named as a gap in waiting, not yet a recurring cost.
- **`DataTable`'s `columnFilters` cannot be read or set from outside the component after mount.** *(P1.)* Found in Delivery Schedule: `sorting`, `globalFilter`, and `pagination` all have a controlled-prop pair; `columnFilters` does not, and is seedable only once via `initialState` at mount. This specifically blocks an externally-triggered quick-filter button ("Today," "This week," "Overdue") — a need that surfaced once, concretely, for a schedule-shaped page, but the underlying API gap would recur identically for any other page wanting the same pattern (a "show me X" button next to, not inside, the table).
- **No exported `DataTableColumnHeader`.** *(P2.)* Named in Procurement Log, reconfirmed unchanged twice. The sortable-header *behavior* works and is used correctly on every page; what's missing is the ability to reuse just that behavior outside `DataTable`'s own internal rendering. An abstraction debt, not a functional one.
- **No in-flight visual state on the rows themselves during a bulk action.** *(P2.)* Named in Procurement Log, reconfirmed in Submittal Log. The pending spinner renders only on the clicked bulk-action button; the affected rows look unchanged throughout, on every page that used bulk actions (all three).
- **No row-click-to-view interaction.** *(P3.)* Named in Procurement Log, reconfirmed in Submittal Log. `TableRow` forwards only native props; every page's "View" action is reachable solely through the row-actions kebab menu.
- **No inline cell editing.** *(P3.)* Named in Procurement Log, reconfirmed in Submittal Log. TanStack Table supports it at the engine level; Krane's wrapper never exposes it. Every page's "Edit" row action is a toast, not a real edit.
- **No column resize or reorder.** *(P3.)* Named in Procurement Log, reconfirmed in Submittal Log. Column *visibility* toggling exists; resize/reorder, also available in the underlying engine, were never wired up.
- **No confirm-before-bulk-action step.** *(P2.)* Named in Procurement Log, reconfirmed in Submittal Log. Every page's destructive bulk actions (Reject, Send back, Flag as delayed) fire immediately with no confirmation `Dialog`, matching the precedent each page deliberately mirrored rather than inventing a new convention.
- **No written guidance for when a categorical table column should render as `Badge` versus plain text.** *(P2.)* Found in Submittal Log: Procurement Log's Status column uses `Badge`; Submittal Log's Stage column (following existing precedent) uses plain text. Both are independently defensible, but the decision rule exists only in each foundation doc's own reasoning, not as one checkable place.

## Forms

- **`Button` has no `loading` prop**, unlike `Input` and `Combobox`. *(P1 — recurring workaround.)* Named in Procurement Log, reconfirmed identically in both later reports — three separate submit buttons (`New purchase order`, `New submittal`, `Schedule delivery`) each had to hand-manage a disabled state and manually swap their own label text ("Creating…"/"Scheduling…") rather than using a consistent, built-in affordance. The single most mechanically-repeated workaround across all three pages.

## Scheduling

- **No calendar, timeline, or Gantt-style date-visualization primitive anywhere in this system.** *(P0 — blocks product development.)* Found in Delivery Schedule, and the single largest-scope gap across all three reports — a missing *category* of UI, not a missing option within an existing one. The only date UI anywhere is the bare native `<input type="date">`. Delivery Schedule built a `DataTable`-based list instead (a legitimate interpretation of "schedule," and the only one buildable from existing components), but this is ranked P0, not P2, for a specific reason the source report could only gesture at: Krane's own existing, already-shipped `Sidebar` names two other destinations — "Project Schedule" and "Lookahead Plan" — whose names imply exactly the visualization this system cannot provide. Those pages haven't been built yet. When they are, this gap will not be a matter of choosing a legitimate alternative interpretation; a project schedule or lookahead plan with no temporal/calendar visualization at all is not a credible substitute the way a sortable delivery list is. This is a scoping decision waiting to block real, named, already-anticipated work.

## Data Visualization

No distinct finding in this category was produced by any of the three audited reports. The closest related gap — the absence of any calendar/timeline primitive — is grouped under Scheduling above, since every report that touched it framed it specifically around date/schedule visualization, not charts, graphs, or general data-visualization needs. No page built so far has attempted or needed a chart, graph, or gauge, so no finding exists to report here. Worth stating plainly rather than padding the category: **this is an open question, not a confirmed gap** — it's possible no Krane workflow ever needs a chart, or possible one does and simply hasn't been built against yet.

## Navigation

`Sidebar`, `Breadcrumb`, and the organization/project switcher chain were confirmed working correctly, with no structural gap, across all three pages and all three viewport widths tested. The one real finding that touches navigation — `SidebarItem`'s `active` state never being announced as the current page — is grouped under Accessibility below, since the defect itself is a missing ARIA attribute, not a structural navigation problem (the *styling* correctly shows which item is active; only the *accessible* signal is missing). No other navigation-specific gap was found in any of the three reports.

## Accessibility

- **`DataTableRowActionsMenu`'s kebab button uses a static `aria-label="Row actions"`** on every row, never parameterized by row. *(P1 — recurring workaround, in the sense that it was found and left unaddressed identically three times.)* Named in Procurement Log, reconfirmed in both later reports, and specifically confirmed in Submittal Log to reproduce **independent of column pinning side** — it is not a side effect of the Tables defect above, but its own, separate gap. A screen-reader user tabbing through any of these pages' tables hears the identical "Row actions" announcement on every row, with no way to tell which row's menu is focused. By contrast, the row-selection checkbox two lines away in the same file *does* get a row-specific label.
- **`SidebarItem`'s `active` prop sets `data-active` for styling but never `aria-current="page"`.** *(P1.)* Named in Procurement Log, reconfirmed identically in both later reports — "Procurement Log," "Submittals," and "Deliveries" were each, in turn, marked active the same styling-only way. `Breadcrumb`, composed on the very same pages, *does* set `aria-current="page"` correctly on its own current segment — two "you are here" signals on one page, one accessible, one not.
- **No skip-to-content link anywhere in `AppShell`.** *(P1.)* Named in Procurement Log, restated as still-unaddressed in Delivery Schedule's carry-forward list. Every page built so far has a five-group, dozen-plus-item sidebar a keyboard user must fully tab through before reaching the actual page content.
- **`RiskIndicator`'s `label` override isn't validated against `level`.** *(P3.)* A pre-existing, already-documented gap (`COMPONENT_STATUS.md`), reconfirmed relevant in Procurement Log and Delivery Schedule (both used `RiskIndicator` on every row); Submittal Log didn't use the component, so didn't re-encounter it. Low severity — the override is rarely used and the mismatch would be visually obvious if it ever happened — but it is a real accessibility question, since `RiskIndicator`'s visible/announced text is exactly what this prop can make misleading.
- **Confirmed working, not a gap, across all three reports**: the `Combobox`-inside-`FormField` integration correctly takes its accessible name from `FormLabel` via `FormControl`'s `aria-labelledby` injection, on every page that used it. Stated here for completeness — the absence of a finding is itself a data point after three independent confirmations.

## Responsive Design

- **`DialogContent`'s own default classes include no `overflow-y-auto`.** *(P1 — recurring workaround.)* Named in Procurement Log, reconfirmed identically in both later reports — three separate forms-in-dialogs (New Purchase Order, New Submittal, New Delivery) each needed the *consumer* to remember to add `overflow-y-auto` manually, since `max-h-[85vh]` alone clips tall content with no scroll affordance. Nothing in `Dialog`'s own API or documentation flags that a tall form needs this; it was discovered fresh, the same way, three times.
- **No page-level action overflow pattern.** *(P2.)* Named once, in Procurement Log: three primary actions (Refresh/Export/New purchase order) already needed `flex-wrap` to avoid clipping at narrow widths. `DataTable` has an established kebab-menu consolidation pattern for *row* actions; nothing equivalent exists for a page's own primary actions. Not yet a recurring cost (`flex-wrap` held up across all three pages, each with two or three actions), but named as the next thing to break once a page needs four or more.
- **Confirmed working, not a gap, across all three reports**: `AppShell`/`Sidebar`/`AppHeader`'s responsive behavior (per `SHELL_RESPONSIVENESS_AUDIT.md`) and `DataTable`'s contained-overflow behavior both held up correctly on every page, at 390/768/1440px, in both light and dark mode, every time they were checked.

## Tokens

- **No `--spacing-*` token scale.** *(P1 — recurring workaround, and the single most-repeated finding across all three reports — named in every one of them.)* Every page's every `gap-2`/`p-4`/`px-3` is a literal Tailwind value chosen ad hoc. Three full pages built on top of this without it ever causing a visible inconsistency — the existing ad-hoc discipline holds — but it has never once been *checkable*, and the absence has now been independently reconfirmed three times with zero new information each time, which is itself the finding: this gap is stable, not improving, and not going to fix itself through more page-building.
- **No sixth-or-later semantic color token.** *(P2.)* Named once, in Procurement Log — the token-level root of `Badge`'s five-variant ceiling above. `--success`/`--warning`/`--destructive`/`--info`/`--pending` is the complete semantic vocabulary in `app/globals.css`; nothing exists to draw from for a conceptually distinct sixth meaning.

## Minimum set of design system improvements required to support Krane's major workflows

Three real workflows now exist as evidence (procurement approval, submittal review, delivery tracking), plus two more already named in the product's own navigation but not yet built against (Project Schedule, Lookahead Plan). Against that evidence, the minimum set was the three P0s plus the P1s that had *already* proven themselves recurring rather than hypothetical — nothing below this line was recommended speculatively. One of the three P0s is now resolved (see "Resolved" below); the remaining items are unchanged.

1. ~~Fix the `DataTable` sticky-column defect.~~ **Resolved.**
2. **Decide, and write down, whether calendar/timeline-shaped views are in scope.** Not "build a calendar" — *decide*. `Project Schedule` and `Lookahead Plan` cannot be built credibly as tables the way Delivery Schedule could; either this gets a foundation doc and an audit the way every other component in this system has had, or someone writes down that it's explicitly out of scope (the way `Card`/`Progress` already are) before a future page-builder re-discovers this exact question from scratch.
3. **Add a permission/role-gating primitive**, or a documented, explicit decision that authorization is handled entirely outside this design system (e.g., server-side, with the UI never rendering privileged actions to begin with — also a legitimate answer, but currently an *implicit* one, not a written one). Every workflow built so far is an approval workflow; none of them can ship to real users in their current, everyone-sees-everything form.
4. **Parameterize `DataTableRowActionsMenu`'s `aria-label`.** Cheap, mechanical, confirmed necessary three times, zero design judgment required.
5. **Add `aria-current="page"` to `SidebarItem`'s `active` state.** Equally cheap, equally confirmed three times.
6. **Add a `loading` prop to `Button`.** Confirmed needed identically three times; the fix is small and the inconsistency it removes (`Input`/`Combobox` already have this) is itself worth closing.
7. **Give `DialogContent` a default `overflow-y-auto`** (or equivalent built-in tall-content handling). Confirmed needed identically three times.
8. **Formalize the `--spacing-*` token scale.** The most-repeated finding in this entire consolidation. Its absence hasn't broken anything yet across three pages — but "hasn't broken anything yet" is exactly the kind of debt that compounds quietly rather than loudly, and three independent confirmations with no new information each time is the signal that it's due, not that it's fine.

Everything else in this document is real and worth tracking, but is either not yet recurring (numeric-range filters, column resize/reorder, inline editing), not yet costly (the `Card`/Drawer absences, both still legitimately deferred per this system's own stated discipline), or genuinely unresolved as an open question rather than a known gap (Data Visualization). None of those block the workflows already proven to matter; the remaining seven items above do, or already have.

## Resolved

### `DataTable` sticky-column rendering defect

**Root cause.** `getPinnedStyle()` in `components/ui/data-table.tsx` positions a pinned column via TanStack Table's `column.getStart("left")` / `column.getAfter("right")`, which sum `column.getSize()` for every column already pinned to the same side. `getSize()` falls back to TanStack's library-wide default of 150px whenever a `ColumnDef` doesn't set `size` explicitly. Neither of `DataTable`'s two internally-generated columns — the select-all checkbox column (`SELECT_COLUMN_ID`) and the row-actions kebab column (`ROW_ACTIONS_COLUMN_ID`) — ever set one. This was invisible in every usage that pins only one column per side (the first item in a pinned-side array always gets `getStart() === 0` regardless of its own size), and only became visible once a page combined `enableRowSelection` with `rowActions`+`positionActionsColumn="first"` — pinning both columns to the same side for the first time anywhere in this codebase, while building Submittal Log.

**Verification before implementing.** Per explicit instruction, real rendered widths were measured live (Playwright, computed styles) before writing the fix, rather than estimated from CSS alone: 28–32.6px (select) and 52–60.8px (actions) in compact density across three real pages, 42.1px and 79.1px in comfortable density on `data-table-advanced`'s Procurement Log table — confirming real per-page variance (likely `table-layout: auto` redistributing slack width), not a single fixed number. A full-repository audit (`grep` across every `enableRowSelection`/`positionActionsColumn`/`rowActions` usage) confirmed all six existing `DataTable` instances pin at most one column per side today, so the fix was guaranteed to be behavior-preserving everywhere it wasn't fixing something.

**Final implementation.** Two new constants, `SELECT_COLUMN_SIZE = 48` and `ACTIONS_COLUMN_SIZE = 88`, added near the existing `SELECT_COLUMN_ID`/`ROW_ACTIONS_COLUMN_ID` constants, each set deliberately above the largest empirically observed width rather than tuned to a tight fit — `getSize()` here only ever feeds the *next* pinned column's offset math, never an actual rendered width, so overshooting costs a few harmless px of extra gap while undershooting reproduces the original bug at a smaller scale. Applied as `size: SELECT_COLUMN_SIZE` / `size: ACTIONS_COLUMN_SIZE` on the two internally-generated `ColumnDef`s. Total diff: 18 lines, entirely within `components/ui/data-table.tsx`, no other file touched.

**Validation.** `tsc`, lint, and `build` all clean. The fix was confirmed working by temporarily reintroducing the exact original trigger condition (`positionActionsColumn="first"` on Submittal Log, which already has `enableRowSelection`) — the actions header's computed `left` changed from the broken `150px` to the correct `48px`, and a full-page screenshot confirmed the "Submittal" header rendering fully legible with no overlap — then reverting that page back to its shipped configuration (confirmed via an empty `git diff` on that file). All six existing `DataTable` usages were re-checked afterward and confirmed to render identically to before the fix (one pinned column per side, `left: 0px`/`right: 0px`, zero change), in both light and dark mode and at both 768px and 1440px, with zero console or page errors throughout.

**Status**: resolved in `components/ui/data-table.tsx`. The select/actions-column-sizing question this raises for any future *consumer-defined* column wanting to pin a third column to an already-occupied side is unchanged — that consumer would need to set its own explicit `size`, exactly as any TanStack Table column author normally would. This was a fix to Krane's two internal columns specifically, not a general "make pinning size-agnostic" change.
