# Krane Design System — DataTable Advanced Foundation

Audit + specification, now implemented at `components/ui/data-table.tsx`, extending the already-shipped v1 in the exact specified order — Pagination, Row Actions, Empty States, Filters, Sticky Columns — demoed at `/demo/data-table-advanced`. `DataTable` shipped as v1 with a deliberately deferred scope, recorded consistently across every status doc since.

One real refinement made at implementation time, not anticipated in this document's own §1 sketch: rather than a single `paginationVariant: "loadMore" | "pages"` prop choosing between presentations, the two shipped as fully independent, opt-in prop groups sharing one underlying TanStack pagination state. A unified switch defaulting to `"pages"` would have made numbered-pagination UI appear on every table that never asked for it — including the pre-existing `/demo/data-table` page — the moment this shipped. Confirmed via Playwright that the original page is completely unaffected. Selection Model (§10) was resolved exactly as recommended (page-scoped selection stays the permanent default; cross-page "select all N" was named as a future, separate, never-default addition once a real paginated/server-driven total exists) but not implemented in this pass — no table in this codebase yet has a server-reported total to make it meaningful. Saved Views shipped as serialization contract only, per explicit instruction — `DataTableState`/`initialState`/`onTableStateChange` exist and are demoed capturing a live snapshot, but `DataTable` itself still has no name, list, or persistence for a "view." See `COMPONENT_STATUS.md`/`DESIGN_SYSTEM_PROGRESS.md` for the full detail.

## Audit: existing codebase state (`components/ui/data-table.tsx`, read in full, not from memory)

- **Bulk Actions — built, solid, one real gap.** `DataTableBulkAction<TData>` (`label`, `icon?`, `variant?`, `onAction`), a `bulkActions` prop, and `DataTableBulkActions` (replaces the toolbar with a `role="region"` bar showing "`N` selected" + action buttons + Clear the moment `enableRowSelection` has any rows selected). The one real gap: `onAction: (rows: TData[]) => void` is synchronous only — a real bulk action ("Approve 12 POs") is realistically a network call, and there's no pending/disabled state while it's in flight, no error surface if it fails. Addressed below.
- **Column Visibility — built, solid, one real gap.** `DataTableColumnVisibility`, a `DropdownMenu` of checkbox items per `column.getCanHide()` column, backed by TanStack's own `VisibilityState`. The real gap: it's local component state, reset on every remount — there's no persistence story, which is exactly the prerequisite Saved Views (below) needs and doesn't have.
- **Empty States — built, but conflates two different reasons a table can be empty.** `DataTableEmptyState` (`title`/`description`, defaulting to "No results") renders identically whether the table has *no data at all* or *has data that the current filter/search matched nothing against*. These call for different messages and different recovery actions (a "create your first PO" CTA vs. a "clear filters" CTA) and currently get the same generic one. A real, found gap, not assumed — addressed below.
- **Loading States — built, reasonably complete.** `isLoading`/`loadingRowCount` render `Skeleton` rows matching the visible column count. No per-row loading treatment for an in-flight mutation on an already-loaded row (e.g., a bulk action processing row by row) — named as a real but lower-priority gap, not built here.
- **Pagination — not built.** What exists instead is a `hasNextPage`/`isFetchingNextPage`/`onLoadMore` "Load more" button — a deliberate v1 choice (a code comment in the file itself: chosen over scroll-triggered loading specifically for its keyboard/screen-reader fallback), not a placeholder for real pagination. The two are different, both legitimate, addressed below — this section is not "fix Load More," it's "decide whether Krane needs page-number navigation alongside it."
- **Row Actions — not built.** No actions column, no per-row menu, confirmed absent in every prior audit and re-confirmed by reading the file directly.
- **Filters — only a single global-search input exists.** `enableGlobalFilter`/`globalFilter`/`onGlobalFilterChange`, wired to TanStack's `getFilteredRowModel`. No per-column filtering of any kind — no filter row, no filter UI, nothing in `ColumnDef` is read for filter configuration. A real, sizable gap, not a refinement of what's there.
- **Saved Views — does not exist as a concept anywhere in this codebase.** No prior doc names it; this document is the first time it's been considered.
- **Sticky Columns — not built.** `DataTable` doesn't set any pinning state and doesn't apply any sticky positioning CSS anywhere.
- **Selection Model — built for the single-page case only, untested against pagination.** `enableRowSelection`, `RowSelectionState`, a header checkbox wired to `toggleAllPageRowsSelected` (TanStack's own page-scoped selection toggle, already — not accidentally — *not* `toggleAllRowsSelected`, the all-pages variant). This was already the right call for a paginate-less v1; it becomes a real, distinct design question the moment real pagination exists, addressed below.
- **The library underneath is already TanStack Table v8** (`@tanstack/react-table`, already a dependency, confirmed in `package.json`), which already ships real, built-in APIs for pagination, column pinning, and column-level filtering. Nothing in this document requires a new dependency — every net-new capability below is "turn on and style a feature TanStack already has," the same position `Tabs` was in relative to `@radix-ui/react-tabs`.

## 1. Pagination

TanStack's own pagination model (`getPaginationRowModel`, `manualPagination`, `onPaginationChange`) is already there to turn on — confirmed directly, not assumed. For server-side pagination, TanStack needs either a `rowCount` (it derives `pageCount` itself) or an explicit `pageCount` (`-1` if genuinely unknown) — both real, documented options, not something Krane has to invent.

**Recommendation: support both presentations on one shared state, not a forced choice.** Atlaskit's `Dynamic Table` defaults to numbered pagination (a `rowsPerPage` prop, pagination controls appear automatically once there's more than one page) — the conventional enterprise-table shape, good for "jump to page 6." The existing `Load more` button is good for a casual, scroll-like browsing feel. Both are legitimate, different UX, and both can sit on the exact same underlying `pagination: { pageIndex, pageSize }` state TanStack already manages — a `paginationVariant: "loadMore" | "pages"` prop chooses the *rendering*, not a second state model. Default to `"pages"` for the enterprise back-office tables this system is actually for; keep `"loadMore"` available since it's already shipped, tested, and has a real accessibility reason behind it.

## 2. Row Actions

Material React Table's API is the cleanest, most directly adoptable reference found: `enableRowActions`, `renderRowActions`/`renderRowActionMenuItems`, and `positionActionsColumn: "first" | "last"`. **Adopted directly, narrowed to Krane's own needs**: a `rowActions: (row: TData) => DataTableBulkAction<TData>[] | undefined`-shaped prop (deliberately reusing the *same* action-item shape `bulkActions` already has — label/icon/variant/onAction — rather than inventing a second, parallel action-description type), rendered as a single kebab (`MoreHorizontal`) button per row opening the already-shipped `DropdownMenu`. `positionActionsColumn` defaults to `"last"` (the common convention every reference system here agrees on) but is real, exposed, not hardcoded.

## 3. Bulk Actions

Already built (audit, above) — this section is the one real addition, not a redesign. `DataTableBulkAction.onAction` becomes `(rows: TData[]) => void | Promise<void>`, and `DataTableBulkActions` tracks an in-flight action: the triggering button shows a spinner and disables its siblings (the same `Loader2`-while-pending pattern the existing `Load more` button already uses, reused rather than reinvented) until the returned promise settles. No new error-surface component — a failed bulk action is exactly what `Toast` already exists for, the consumer's own `onAction` catches and reports it, the same separation `Stepper` keeps from `Form`'s validation.

## 4. Column Visibility

Already built and sufficient as a UI (audit, above). The only real addition this section makes is *not* inside this feature — column visibility becomes one of the fields Saved Views (§6) serializes, not a change to how visibility itself works.

## 5. Filters

The real, sizable gap named in the audit. Material React Table's `filterVariant` (`text`/`select`/`multi-select`/`range`/`range-slider`/`date`/`datetime`, declared per column) is the directly adoptable reference shape — a column declares *how* it should be filtered, the table renders the right control automatically. **Narrowed deliberately for Krane**: three variants, not seven — `text`, `select`, and `dateRange` — matching the actual column shapes every named Krane table has (a free-text column, a fixed-enumeration column like Status/Vendor, a date column), not MRT's full surface, the same "narrower than the reference, not because it's wrong, but because nothing here needs the rest" judgment already applied when `Stepper`'s family came out smaller than Material UI's own `Step`/`StepIcon`/`StepButton` split. Declared via `ColumnDef.meta.filterVariant` (TanStack Table's own typed extension point for exactly this), rendered in a filter row reusing already-shipped controls directly — `Input` for `text`, `Select` for `select` (the column's own distinct values, or a consumer-supplied option list for a fixed enum like Status), two `Input type="date"` (or a future date-range control, not built yet) for `dateRange`. No new filter-input primitive — every control this needs already exists.

## 6. Saved Views

Linear's own implementation is the load-bearing precedent: a filtered list becomes a named, reusable view via a single "Save view" action that appears once at least one filter is active — confirming a "view" is a *bundle*, not a separate concept: filters + sort + display options (Krane's equivalent: column visibility + density), saved together. **The real architectural decision, not a feature to build inside `DataTable`**: this system's own, repeated discipline — `Stepper` doesn't own validation, `Tabs` doesn't know `Form` exists, `AccountMenu` doesn't own where "Settings" actually goes — applies here directly. `DataTable` itself has no concept of a "saved view," no name, no persistence, no list of a user's views. What it exposes instead is a single, serializable snapshot — `{ sorting, columnFilters, columnVisibility, density, globalFilter }` — via a `tableState`/`onTableStateChange` pair (mirroring the controlled-prop shape `sorting`/`onSortingChange` already established) and an `initialState` prop to *apply* a previously-saved snapshot back. Naming, storing, and switching between named views is a separate, future, product-level concern (most plausibly a `ViewSwitcher` component, explicitly out of this document's scope) built *on top of* this snapshot, not inside `DataTable`.

## 7. Empty States

The real, found-during-audit gap: "no data" and "no results from your filter" are currently the same message. **Two distinct, named empty states, not a richer single one**: `DataTable` already knows which case it's in — `data.length === 0` (true empty) vs. `rows.length === 0 && data.length > 0` (filtered/searched down to nothing) — and can pick a sensible default for each (a quieter "No results match your filters" + a "Clear filters" action for the second case, distinct from a "No \[entity\] yet" first-case default) while still honoring a fully custom `emptyState` override for either, exactly as today.

## 8. Loading States

Already built and sufficient (audit, above). Material React Table's loading *overlay* (dimming already-rendered rows during a refetch, as an alternative to swapping in skeleton rows) is named here as a real, citable alternative pattern — not adopted, since Krane's existing row-skeleton approach is already shipped, tested, and visually consistent with `Skeleton`'s one other use in this system; named so a future contributor doesn't reach for it without knowing it was a considered, not overlooked, alternative.

## 9. Sticky Columns

TanStack Table's own column-pinning API — `column.pin("left" | "right")`, `getIsPinned()`, `getStart()`/`getAfter()` for the actual sticky CSS offset, `getIsFirstColumn()`/`getIsLastColumn()` for an edge box-shadow — is already part of the engine `DataTable` is built on, confirmed directly, not assumed. **Recommendation**: pin the existing selection column left and the new row-actions column (§2) right, by default, whenever each is enabled — the two columns most likely to need to stay reachable while a wide table scrolls horizontally, and the two that have no business value sitting deep within scrollable content the way an ordinary data column does. No general-purpose "pin any column" UI in v1 — the two structural columns are the named, concrete need; an arbitrary user-pinned data column is real (TanStack supports it natively at zero marginal cost) but has no named Krane use case yet.

## 10. Selection Model

The single most important finding in this entire document, and the one area where getting the *easy*-seeming version wrong is a documented, real mistake, not a hypothetical: Shopify Polaris's own `IndexTable` has a confirmed, filed issue where a paginated table's bulk-action "select all" defaulted to selecting *every row across every page* instead of just the current page — exactly the kind of surprising, silent-data-scope mistake a bulk "Approve" or "Delete" action must never make. Polaris's own documented fix is the pattern this document adopts directly: **selection always defaults to the current page only** (`toggleAllPageRowsSelected`, already what Krane's v1 correctly uses); a *separate*, explicit, secondary affordance — "Select all `N` matching rows" — appears only once a known total across the full filtered result set exists (i.e., once real pagination or a server-reported count is wired up) and is never the default, always a second, deliberate click. This directly resolves the open question the audit named: page-scoped selection was already the right instinct for a paginate-less v1, and this section makes the *future*, post-pagination behavior an explicit, documented decision instead of something a future contributor would have to rediscover the hard way Polaris's own users apparently did.

## Comparison: TanStack Table, Atlassian Dynamic Table, Polaris IndexTable, Linear, Material React Table

**TanStack Table** — the engine `DataTable` is already built on. Confirmed, not assumed: real `getPaginationRowModel`/`manualPagination` (§1), real column-pinning APIs (§9), and a typed `ColumnDef.meta` extension point exactly suited to declaring per-column filter behavior (§5). [(TanStack — Pagination)](https://tanstack.com/table/v8/docs/guide/pagination) [(TanStack — Column Pinning)](https://tanstack.com/table/v8/docs/guide/column-pinning) **Verdict**: the single most load-bearing comparison in this document, because every net-new capability above is "turn on what's already in the library," not new engineering risk.

**Atlassian (Atlaskit) Dynamic Table** — pagination toggled by a single `rowsPerPage` prop; sorting/pagination/loading/drag-reorder state all managed internally by default. [(Atlaskit — Dynamic table)](https://atlassian.design/components/dynamic-table) **Verdict**: confirms numbered pagination as the conventional default worth offering (§1); drag-to-reorder rows is real and explicitly not adopted here — no named Krane use case for reorderable rows today.

**Shopify Polaris (IndexTable)** — ships a real "bulk actions and selection across pages" example, and has a real, filed, confirmed bug where naive select-all-with-pagination selected everything instead of the current page. [(Polaris — Index table)](https://polaris-react.shopify.com/components/tables/index-table) **Verdict**: the single most important finding in this document — directly resolves §10, not just an inspiration for it.

**Linear** — filters apply broadly across views; a "Save view" action persists the current filter + display-option combination as a named, reusable view, appearing only once a filter is actually active. [(Linear Docs — Custom Views)](https://linear.app/docs/custom-views) [(Linear Docs — Filters)](https://linear.app/docs/filters) **Verdict**: the direct source for §6's "a view is a bundle" framing, and for the architectural boundary (Linear's own views are an *application* feature built on top of its filtering, not folded into the filter UI itself) — adopted directly rather than rediscovered.

**Material React Table** — `filterVariant` per column (§5, narrowed for Krane); `enableRowActions`/`renderRowActions`/`positionActionsColumn` (§2, adopted directly); a loading overlay as an alternative to skeleton rows (§8, named, not adopted). [(MRT — Column Filtering)](https://www.material-react-table.com/docs/guides/column-filtering) [(MRT — Row Actions)](https://www.material-react-table.com/docs/guides/row-actions) **Verdict**: built on the same TanStack core `DataTable` already uses, making it the most directly transferable API shape of all five — confirmed by how many of its props this document adopts close to verbatim, narrowed rather than reinvented.

## Existing capabilities

Bulk actions, column visibility, two empty-state inputs (now correctly distinguished), row-skeleton loading, and the global-filter/sort architecture, all already shipped and audited above as sufficient or near-sufficient. The underlying TanStack Table engine already has pagination, column-pinning, and per-column-filter APIs ready to be turned on — none of this document's net-new scope requires a new dependency.

## Missing capabilities

Pagination in the numbered-pages sense (not just `Load more`). A row-actions column. Per-column filters of any kind. Any concept of a saved view — not even named anywhere before this document. Sticky/pinned columns. A post-pagination selection model — not missing exactly, but genuinely untested and undecided, which this document resolves before it becomes a real, live footgun the way it was for Polaris's own users.

## Krane DataTable Advanced architecture (recommended)

Ten decisions, each scoped to what's actually needed rather than each reference system's full surface: both pagination presentations on one shared TanStack pagination state, defaulting to numbered pages; a row-actions column reusing the exact same action-item shape `bulkActions` already has; an async-aware bulk-action callback; three filter variants (not seven) declared via `ColumnDef.meta`; a serializable table-state snapshot for Saved Views to be built *on top of*, not inside; two distinct empty-state defaults instead of one generic one; the existing loading treatment kept as-is; the selection and actions columns pinned by default via TanStack's own column-pinning API; and a selection model that defaults to page-scoped forever, with cross-page "select all `N`" as an explicit, separate, never-default action.

## Final recommendation for Krane

Build all ten as direct extensions of the already-shipped `DataTable`, not a rewrite — every new capability is additive to the existing prop surface (`paginationVariant`, `rowActions`, `tableState`/`onTableStateChange`, `columnDef.meta.filterVariant`), and every one of them turns on a capability the underlying TanStack Table engine already has rather than introducing new engineering risk. Treat §10 (Selection Model) as the one section worth re-reading before implementation begins — it's the one place a reference system's own real, filed bug shows exactly what happens when the "obvious" version is built first and the deliberate version is treated as a later refinement.
