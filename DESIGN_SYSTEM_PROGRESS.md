## 4. Known technical decisions

- **DataTable owns its TanStack Table instance internally** rather than exposing a `useDataTable` hook — simpler v1 API, traded against consumer flexibility on purpose.
- **Toast required an external store** (`hooks/use-toast.ts`) because Radix's Toast API is declarative, not imperative — `Toast`/`Toaster`/`ToastProvider` alone can't be called as `toast("message")` from arbitrary code without it.
- **`TooltipProvider` and `Toaster` are mounted once at the root layout**, not per-page — required for `TooltipProvider`'s shared delay-grouping behavior to mean anything, and for `Toaster` to be reachable from any route.
- **Geist is self-hosted via `next/font/local`**, not `next/font/google` — the provided font package included the full variable weight axis (100–900) plus a separate italic variable file; no new npm dependency, since `next/font/local` ships inside `next` itself.
- **The global `--tw-duration`/`--tw-ease` motion override was deliberately not implemented** after discovering it would silently regress the Collapsible system's own distinct fallback — explicit per-component classes were used on the five floating-UI primitives instead, at the cost of more files touched per future component.
- **Two of `Input`'s native HTML attributes (`size`, then later `prefix`/`suffix`) collided with same-named custom props** — both discovered via real `tsc` failures during implementation, not by inspection, and both resolved by `Omit<>`-ing the native attribute. Worth remembering as a pattern: any future prop name should be checked against `React.ComponentProps<"input">`'s actual native attributes before assuming it's free to use.
- **Every token family follows the same two-layer pattern**: a raw, mode-dependent value (`:root`/`.dark`) plus a stable Tailwind-facing registration (`@theme inline`) — the mechanism that lets one utility class mean something different per theme mode without `@theme inline` needing two definitions. Applied without exception across colors, and skipped (correctly) for mode-independent families like motion/z-index/typography, which live directly in `@theme inline` as literal values.
- **Every new Tailwind v4 theme namespace used in this project (`--text-*`, `--shadow-*`, `--duration-*`, `--ease-*`, `--z-*`) was confirmed empirically** by inspecting compiled CSS output, not assumed from documentation — this caught real surprises twice (shadow/text utilities not appearing until something actually used them; the native-attribute collisions above).
- **`lib/utils.ts`'s `cn()` silently dropped `text-label` when combined with a text-color class** (`cn("text-label", "text-foreground")` rendered as just `text-foreground`) — found while building `Label`, which combines exactly that pair. Root cause: `tailwind-merge` doesn't know about Krane's custom `text-{role}` typography scale, so it filed all six roles under its built-in `text-color` group, where they conflicted with real color utilities. Fixed by switching to `extendTailwindMerge` and registering `text-display/heading/title/body/caption/label` under the `font-size` group instead — a third instance of the "confirmed empirically" pattern above, this time at the `cn()`/runtime-merge layer rather than compiled CSS. Any future component pairing a typography-role class with a color class would have hit the same silent drop.

## 5. Known design decisions

- **Status (operational state) and Risk (severity) are distinguished by component form, not hue** — Badge stays a solid pill, RiskIndicator stays an outlined chip + dot meter, deliberately, after relocating either palette was tried and rejected as worse for users' learned green/red conventions.
- **Krane Yellow is withheld from anywhere it could be mistaken for Warning/Pending** — most concretely, DataTable's selected-row tint uses the previously-unused `--accent` token instead of brand yellow, specifically to avoid a selected row reading as an at-risk one.
- **`--accent` and `--sidebar-accent` were both given their first real jobs** rather than minting new tokens — selected-row tint and the Shell's active/hover state, respectively.
- **Dialog uses `bg-card`; Popover/DropdownMenu/Tooltip use `bg-popover`** — content panel vs. transient floating chrome is treated as a real distinction, not collapsed into one surface token.
- **Shell tokens (the navy sidebar/header) never flip with light/dark mode** — a deliberate brand constant, not an oversight in an otherwise mode-aware system.
- **App Shell nav groups are collapsible** — reversing an explicit earlier instruction that they should always stay expanded, once a later instruction explicitly asked for the opposite. Current behavior reflects the most recent direction given.
- **`Input` has no color/style variants** — one visual treatment for every text field, on the reasoning that a field's *intent* never varies the way a Button's does. Its size scale also deliberately doesn't align 1:1 with `Button`'s (Input's `default` height matches Button's `lg`), because two already-shipped real call sites had independently converged on that height before the scale was formalized.
- **`Label` gets no size variant**, matching all four externally-reviewed reference systems, and its read-only state produces **no visual change at all** — read-only content is still "real," unlike disabled content, and shouldn't look turned off. Its `disabled` prop is an explicit boolean, not a `peer-disabled:` selector, specifically because this codebase's dominant layout puts the label before the control (see `LABEL_FOUNDATION.md` §5/§7 for the full correction).
- **`Textarea` uses `text-body`, not `Input`'s bare `text-sm`** — the first real consumer of that typography role as a form-control style rather than prose; `text-body`'s defining feature (1.5 line-height) only matters once content actually wraps, which `Input`'s single-line text never does. Also gets no size-tier prop (no `sm`/`default`/`lg`) — `min-h-[80px]` plus native `rows` passthrough is the sizing lever instead, matching the precedent of every reference system that ships a standalone Textarea (neither gives it a density-tier prop the way their single-line field gets one).
- **`Textarea`'s resize behavior was decided by auditing real product/library behavior, not just reference-system APIs** (`TEXTAREA_RESIZE_AUDIT.md`) — shipped as `resize-y` (vertical only), explicitly as a deliberate stepping stone toward auto-grow rather than an end state. GitHub ships its own official auto-grow override package on top of Primer's `resize: both` default; Atlaskit's current-generation `Textarea` defaults to auto-grow (`smart`) directly; Linear's editor architecture (contentEditable, not a plain textarea) has no manual resize handle at all. Auto-grow was still deferred because `field-sizing: content` lacks Safari/Firefox support today and no concrete Krane consumer has asked for it yet — same bar applied to every other deferred feature in this system. Read-only additionally drops the resize handle (`read-only:resize-none`) — a real correction over `Input`'s read-only treatment, which has no resize concept to correct.
- **`Select` is deliberately scoped to bounded, fixed enumerations only** (CSI division, status/category pickers) — `SELECT_FOUNDATION.md` found that every mature reference system audited (Material, Atlaskit, Polaris, Primer) independently ships a *separate* searchable component for open-ended entity catalogs rather than stretching one Select to cover both shapes of list, and Krane follows the same split. Vendor/manufacturer/material/project/user pickers are explicitly **not** `Select`'s job — they're `Combobox`'s, built immediately after (§2/§6).
- **`Select` gets `Input`'s exact size scale** (`sm`/`default`/`lg`, not its own), the opposite call from `Textarea` — Select's trigger is fixed-height and single-line, the same shape as Input's content, so it inherits Input's tiers verbatim rather than omitting a size axis the way Textarea did.
- **`Select`'s `readOnly` is a Krane-specific construct, not a pass-through** — neither the native `<select>` element nor Radix `Select` has a read-only concept (MDN: `readonly` only applies to text-like controls). Implemented as `disabled` for interaction-blocking, styled with `Input`/`Textarea`'s read-only treatment (full opacity, `bg-muted`, `cursor-default`) rather than the dimmed disabled look, **with the chevron icon hidden entirely** — a control that can't be opened shouldn't display the affordance that says it can.
- **`Combobox` is built on `cmdk` composed with the already-shipped `Popover`, not `cmdk`'s own `Command.Dialog` and not a hand-rolled `Popover`+`Input` filter** — `Command.Dialog` is `cmdk`'s modal/elevated command-palette shape, wrong for an inline field; a hand-rolled filter would mean re-deriving the real WAI-ARIA combobox pattern (`aria-activedescendant` tracked against a live-filtered list) from scratch, which `cmdk` already implements correctly (confirmed empirically against its shipped source: real `role="combobox"`/`"listbox"`/`"option"` and `aria-activedescendant` wiring, not assumed).
- **`Combobox`'s loading and empty states are explicitly gated against each other in the JSX** (`loading ? <Command.Loading> : <>...<Command.Empty>...</>`), never left to render simultaneously — `cmdk`'s own issue tracker has an open report of exactly that double-render happening when left to chance, found during this project's own pre-implementation audit (`COMBOBOX_FOUNDATION.md` §6) rather than discovered as a bug afterward.
- **This is the first new runtime dependency added to this project since its initial scaffold.** Every component before `Combobox` — Checkbox through Select — was built by wrapping an already-installed primitive (`radix-ui`) or a native element. `cmdk` breaks that streak deliberately, not by accident: `SELECT_FOUNDATION.md`/`COMBOBOX_FOUNDATION.md` evaluated and rejected the zero-new-dependency alternative first, and the addition was made only on an explicit instruction naming `cmdk` directly — the same bar later applied to `react-hook-form`/`zod`/`@hookform/resolvers` when the `Form` layer's turn came (§6).
- **`RadioGroup`'s `status` vocabulary lives on the group root, not on each `RadioGroupItem`** — a deliberate departure from the original `FORM_SPECIFICATIONS.md` spec's "invalid, applied per-item" framing, made for the same reason `Select`'s `status` sits on the trigger rather than on each `SelectItem`: a radio group is one field, and "this field is invalid" should color every option consistently, not leave the consumer to repeat `status="error"` on each item by hand. Propagated via React context (the same mechanism Radix's own `RadioGroup.Root`/`Item` composition already uses internally for `value`/`disabled`/`name`), with a per-item override left available for the rare case the original spec actually had in mind. `aria-invalid` is set on the group itself, not each item — confirmed safe before relying on it: `aria-invalid` is one of WAI-ARIA's global attributes (unlike `aria-readonly`, which is role-restricted and is why `Combobox`'s trigger doesn't carry it — see above), so a `role="radiogroup"` container supporting it isn't a guess.
- **`RadioGroup`'s hand-rolled stand-in inside the `Label` demo (built before a real `RadioGroup` component existed) was migrated to the real component** the moment it shipped — the same closure already applied when `Label` itself shipped and the three hand-rolled `<label>` elements in the `Input` demo were migrated.
- **`Switch`'s `status` recolors only the ring, never the on/off track fill** — unlike `RadioGroup`, `Switch` is a single control with no root/item split to put `status` on, so it sits directly on the one element; but the track's `bg-input`/`bg-primary` distinction is the control's primary, highest-traffic signal ("is this on"), and overloading it with a second meaning ("is this valid") would make the two questions visually collide. Error keeps the persistent (non-focus-gated) ring `FORM_SPECIFICATIONS.md` originally specified for `aria-invalid`; success/warning generalize that into focus-only reinforcement, the same quieter treatment already used for `Input`/`Select`/`RadioGroup`.
- **`Switch`'s track needs `border-2 border-transparent`, not just `h-5 w-9`**, for the spec's own described geometry (`translate-x-0` flush left, `translate-x-4` flush right) to actually land correctly — a detail the prose spec implied but didn't spell out; without it the thumb overshoots the track on one side. Confirmed by checking the rendered output, not assumed from the written spec.
- **`Form`'s `FormControl` injects `aria-labelledby`, not just `id`, onto every wrapped field** — a real addition beyond `FORM_FOUNDATION.md`'s documented architecture, found by actually verifying (not assuming) that all seven field primitives work correctly inside `FormControl` before building it. `RadioGroup`'s root is a `<div role="radiogroup">`, not a native labelable element, so `FormLabel`'s standard `htmlFor` association would have been completely inert for it — no accessible name at all, not just a degraded one. `aria-labelledby` fixes this by construction, and is harmless for the other six primitives: ARIA's accessible-name computation gives `aria-labelledby` precedence over a native `for`/`id` label, but it resolves to the exact same label text either way, and native label click-proxying is governed by `for`/`id` matching alone, unaffected by ARIA naming.
- **`Combobox` needed a small, pre-named API extension before `Form` could wrap it** — `aria-describedby`/`aria-labelledby` passthrough and an overridable `aria-invalid`, exactly the prerequisite `FORM_FOUNDATION.md` flagged during its own audit, not discovered mid-build. `aria-invalid` itself ended up *not* rendered as a literal attribute on the trigger `<button>` — lint confirmed the implicit `button` role doesn't support it, unlike `switch`/`radiogroup`/`combobox` roles, which all do. Adding `role="combobox"` to fix that would have then required statically-visible `aria-controls`/`aria-expanded` Radix only sets at runtime — not worth chasing; the computed value is exposed as `data-invalid` instead, and the real signal still reaches users via `aria-describedby` (pointing at `FormMessage`'s `role="alert"` text) plus the untouched visual `status` styling.
- **`amount` in the `Form` demo's schema deliberately avoids `z.coerce.number()`** — combining zod v4's coercion with `react-hook-form`'s generics produced a real, verified type mismatch (the resolver's input type becomes `unknown` pre-coercion, conflicting with `useForm<T>`'s single declared type). Resolving it correctly means splitting `useForm` into `z.input`/`z.output` generics — real complexity for a demo. Simpler, equally correct fix used instead: plain `z.number()` in the schema, with the `Input`'s `onChange` explicitly calling `event.target.valueAsNumber` rather than spreading `{...field}` blindly — the standard, common pattern for numeric `react-hook-form` fields regardless of validation library.
- **`Alert` reuses `Toast`'s exact variant recipe as a styling convention, not an import** — `border-{color}/30 bg-{color}/10 text-{color}-text` per variant, the same relationship `RadioGroup` already has to `Checkbox`'s selection-color choice. But `AlertTitle`/`AlertDescription` deliberately do **not** copy `Toast`'s own `ToastTitle`/`ToastDescription` typography (`text-sm font-semibold`/`text-sm opacity-90`) — `Toast` predates this project's `text-label`/`text-body` discipline (established later, during `Label`/`Textarea`), and `Alert` uses the formal tokens from the start rather than repeat the same drift a second time. `Toast` itself was left unchanged — out of scope for this pass, a known, documented drift, not a silent inconsistency.
- **`Alert`'s `role` defaults to `region` (with a variant-derived `aria-label`, e.g. "Warning"), never `role="alert"` driven by variant** — confirmed via MDN before relying on it: `role="alert"` is specifically for content that's dynamically displayed, not content already present on page load, so hardcoding it to the `error` variant would have been wrong for the common case (a `warning`/`error` banner that's simply part of the page from the start) and only coincidentally right for the rare one (a validation summary appearing right after a failed submit). `role` is an explicit, opt-in prop instead, demoed in the `Form`-composition example where it's actually warranted.
- **`Alert`'s dismiss and action props are singular (`onDismiss`, `primaryAction`, `secondaryAction`), not an array** — `ALERT_FOUNDATION.md`'s "at most one primary and one secondary action" guideline (independently converged on by both Shopify Polaris and Atlassian's `SectionMessage` documentation) is enforced by the type itself, not left to a code-review convention that could be missed.
- **`Alert` ships as three components (`Alert`/`AlertTitle`/`AlertDescription`), not six** — the foundation doc's own API sketch loosely suggested separate `AlertActions`/`AlertClose`/`AlertAction` sub-components, but explicitly hedged that sketch as "documentation-stage, not a frozen contract." The narrower, prop-driven shape was the actual build instruction, and is simpler for the common case (most alerts have zero or one action) without losing any documented capability.
- **The theme switcher is hand-rolled, not `next-themes`** — a deliberate departure from shadcn's own canonical answer for a shadcn-convention project, made because the actual problem (a binary light/dark/system toggle, persisted, no-flash) is genuinely smaller than the bar this project has applied to every other new-dependency decision (`cmdk`'s real WAI-ARIA combobox complexity, `react-hook-form`'s full validation state machine). Reaching for a dependency here would have been the first time a new one was added *without* that justification.
- **Built on `React.useSyncExternalStore`, not `useState`+`useEffect`** — found necessary, not chosen upfront: an initial `useEffect`-based read of `localStorage` on mount hit a real `react-hooks/set-state-in-effect` lint error (calling `setState` synchronously in an effect body risks cascading renders). `useSyncExternalStore` is the React-correct tool for "read a value an SSR render can't see" — it renders a server-safe snapshot during SSR and the first client render, then synchronously corrects to the real client value, with no setState-in-effect at all. A second, unplanned benefit: subscribing to the native `storage` event for this (needed anyway, since the same-tab case requires its own pub-sub — `storage` only fires in *other* tabs) gives cross-tab sync for free, a capability `THEME_SWITCHER_FOUNDATION.md` had named as a deferred nice-to-have after seeing it in MUI's `colorSchemes` API.
- **`AppShell`'s main content panel was hardcoded `bg-white`, found during the theme switcher's pre-implementation audit, fixed in the same pass** — defensible while dark mode was unreachable (the "framed white card" look was a deliberate choice, not an oversight), not defensible once a real toggle exists: every token-driven component inside it would have flipped to dark colors while sitting on a panel that stayed forced-white. Now `bg-background text-foreground`. Verified by a full Playwright pass, not just by re-reading the class name: computed-style reads (`<main>`'s `background-color`: pure white in light, near-black in dark) plus screenshots across `/demo/app-shell` and a 6-page dark-mode spot-check, finding zero other token issues anywhere in the system.
- **Swaps suppress CSS transitions for one frame** (a scoped `.no-theme-transition` class in `app/globals.css`, applied and removed by the same function that toggles `.dark`, never left in the DOM) — both `next-themes` (`disableTransitionOnChange`) and MUI's `colorSchemes` document this as a real, named requirement, not an edge case: an instant swap avoids an animated full-viewport color sweep that's unpleasant for everyone and a genuine concern for motion-sensitive users.
- **`Avatar`'s status dot reuses the existing `success`/`warning`/`error`/`info` vocabulary, not a new presence enum** — Atlaskit's own documentation draws a real distinction between *presence* (online/away/offline, a live-collaboration concept) and *status* (an action's outcome, like approve/decline). Krane's procurement domain has a concrete use for the second and none for the first, so only the *reasoning* was adopted, not an online/away/offline vocabulary with no other foothold anywhere in this system's domain model.
- **The status dot is a sibling of `Avatar`'s Radix `Root`, not a child of it** — `Root` needs `overflow-hidden` to crop an image into its circle/square shape, and a corner-positioned dot placed inside that container would be clipped at the edge. Both are wrapped in one outer `relative` span instead, found necessary while implementing, not anticipated in `AVATAR_FOUNDATION.md`'s own prose.
- **`AvatarFallback` does not default `delayMs`** — a real bug found and fixed during implementation, not assumed safe from the foundation doc's own recommendation to use it. Reading Radix's source directly showed `canRender` starts `false` whenever `delayMs` is defined *at all*, regardless of whether an `AvatarImage` sibling exists to wait for — an unconditional `delayMs={300}` would have meant every plain initials/icon avatar rendered nothing for 300ms. Fixed by leaving `delayMs` a plain pass-through prop (Radix's own `undefined` default renders immediately); the demo passes it explicitly only on the two examples that pair `AvatarFallback` with a real `AvatarImage`.
- **`AccountMenu` is filed under `components/shell/`, not `components/ui/`** — chrome with fixed, Krane-specific structure (identity/actions/context/sign-out, in that order, no consumer-configurable layout), the same category `OrganizationSwitcher`/`ProjectSwitcher` are already in, not a generic primitive a future page would recompose differently.
- **`organizationLabel` is required, not optional, refining `ACCOUNT_MENU_FOUNDATION.md`'s own API sketch** — every real Krane session already has an organization context (`OrganizationSwitcher`'s own existence presumes one), so hedging it as optional would have modeled a scenario that doesn't actually occur in this domain.
- **Profile/Settings/Notification Preferences ship as three distinct, always-rendered rows, not two** — the foundation doc's own sketch bundled "Profile settings" into one row; explicit instruction at implementation time split it into three, the same shape of refinement `Alert`'s six-component API sketch already got narrowed at its own implementation time. The rows themselves are fixed structure; only their `onClick` handlers are optional.
- **`ThemeToggle` and `OrganizationSwitcher` were deliberately not folded into `AccountMenu`** — both stay separate, already-shipped, already-justified controls, exactly as `ACCOUNT_MENU_FOUNDATION.md` recommended after finding GitHub's and Linear's own architectures keep the same separation rather than consolidating for its own sake.
- **`Tabs`'s active indicator uses the explicit `data-[state=active]:` syntax, not a `data-active:` shorthand** — found and resolved during the foundation pass, not assumed safe from the working `data-open:`/`data-closed:` precedent already used by `Dialog`/`DropdownMenu`/`Popover`/`Toast`. Checked directly against the real compiled stylesheet: Tailwind v4 special-cases `open`/`closed` specifically, generating a rule that matches *either* `[data-state=open]` *or* a bare `[data-open]` attribute — confirmed by inspecting `.next/static/chunks/*.css` directly, not inferred. Nothing guarantees an equivalent special case exists for an arbitrary `active`/`inactive` pair that's never been referenced in source, so the explicit, universally-correct `data-[state=active]:` form was used instead of gambling on an unconfirmed shorthand.
- **`activationMode` passes straight through to Radix as a per-instance prop, not a value the wrapper defaults differently from Radix's own `automatic`** — `TABS_FOUNDATION.md`'s own framework (automatic when content is already loaded, manual when a tab triggers its own expensive fetch) is a decision each call site makes, not something `Tabs` itself should bake in either direction.
- **No overflow handling, no animated sliding indicator, `horizontal` orientation only** — all three named explicitly in `TABS_FOUNDATION.md` as deferred without a concrete Krane consumer, not discovered as gaps during implementation.
- **`Breadcrumb`'s current-page segment is derived from array position, never an explicit flag** — GitHub Primer's own reference API requires the consumer to manually mark the last item `selected`; `BREADCRUMB_FOUNDATION.md` found this a real, avoidable footgun (forget the flag, or set it on the wrong item after a list is reordered, and the current page silently renders as a clickable link to itself with no `aria-current`). Krane's version makes this impossible to get wrong by construction — the last item in the array is always the non-interactive, `aria-current="page"` segment.
- **`Breadcrumb` deliberately does not merge with, or get built on top of, `AppHeader`'s existing organization/project switcher chain**, despite both using the identical `ChevronRight` separator and looking alike at a glance — the switcher chain is global, persistent chrome for *changing* context (a `DropdownMenu` per segment); `Breadcrumb` is page-level, mostly-static content for *showing* hierarchy and going *up*. Verified both can coexist on one real page (`/demo/app-shell`) without redundancy, not just argued in prose.
- **Shopify Polaris's single-`backAction` simplification (no multi-level breadcrumb at all) was considered and explicitly not adopted** — Krane's real entity hierarchies (Vendor → Contract, Org → Project → Submittal) are genuinely multi-level, and collapsing a trail to "go back one step" would lose information a full trail correctly preserves.
- **No collapse/truncation for long breadcrumb trails in v1** — Material UI's `maxItems` and Atlaskit's default-8-item threshold are both real, converging precedents, but Material UI's own issue tracker has a confirmed keyboard-accessibility bug in exactly that feature (the collapsed ellipsis isn't reachable via keyboard) — named as a specific reason to build this carefully whenever a real long trail appears, not a reason to avoid it forever.
- **`Stepper` is two layers, not one** — `StepperList`/`StepperStep` (the indicator) is usable completely standalone, with no `Stepper` wrapper, for a read-only progress tracker; `Stepper`/`StepperContent` adds the interactive wizard machinery (Next/Back, gated linear progression) on top of the same indicator. Found necessary, not a stylistic split: Atlaskit's own documentation names this one pattern five different things ("Progress tracker, Stepper, Steps, Timeline, Meter"), independent confirmation that one visual pattern genuinely serves two different jobs industry-wide, not a Krane-specific overthink.
- **`Stepper` is built on no Radix primitive, deliberately not `Tabs`** — WAI-ARIA has no stepper role the way it has `tab`/`tabpanel`, and `Tabs`' own activation model (switch on focus, or on `Enter`/`Space`) has no concept of "you can't reach this step yet," the wrong shape for gated, linear progression.
- **Advancing a step moves real DOM focus to the new step's heading — the one hard, non-negotiable requirement of the whole component**, sourced directly from W3C's own multi-step-forms guidance ("move focus to the starting point for that action"), not a generic SPA-navigation assumption. Verified via Playwright after `Next`, `Back`, and indicator-click navigation, and explicitly verified *not* to fire on initial mount (which would steal focus from normal page-load behavior for no reason).
- **`upcoming`/`current`/`completed` are derived from position; `error` is the one explicit prop** — the same "derive from position, don't require a flag" discipline `Breadcrumb` already established, with `error` correctly excepted because no position-based derivation could ever know a completed-looking step is actually invalid; that knowledge belongs to the consumer's own form-validation state.
- **`Stepper` owns position and navigation only, never validation** — `Form`'s already-shipped `react-hook-form`+`zod` machinery is composed in by the consumer's own "Next" handler, the same separation already established between `Tabs` and `Form`.
- **Step labels hide below the `sm` breakpoint — the first real responsive treatment anywhere in this system**, not a broader mobile-shell effort. `FINAL_AUDIT.md` had flagged mobile/responsive behavior as unimplemented everywhere; this is a narrow, component-scoped exception (the active step's label is never actually lost, since `StepperContent`'s own "Step X of Y" heading still renders it as real text regardless of viewport), not a start on the still-unaddressed App Shell responsive item.
- **DataTable's two pagination presentations are entirely independent, not one prop deciding between them** — the foundation doc's own sketch suggested a single `paginationVariant: "loadMore" | "pages"` switch, but that risked a real backward-compatibility regression: defaulting it to `"pages"` would have made numbered-pagination UI appear on every existing table that never asked for it, including the already-shipped `/demo/data-table` page. Implemented instead as two fully independent, opt-in prop groups (`hasNextPage`/`onLoadMore` unchanged; `enablePagination`/`pagination`/`onPaginationChange` net-new) sharing the same underlying TanStack pagination state shape — a deliberate, found-during-implementation refinement, confirmed via Playwright that the pre-existing demo is pixel-for-pixel unaffected.
- **Row actions reuse `DataTableBulkAction`'s exact shape rather than a parallel type** — `onAction` is called with a single-element array even for a single row, per the foundation doc's own explicit instruction, so bulk and row actions are interchangeable wherever an action list is defined once and used in both places.
- **Pinned columns get a flat background regardless of row hover/selection state** — a deliberate, accepted simplification named directly in the code: matching a sticky cell's background to its row's actual hover/selected tint would require either a `group`-based Tailwind pattern or per-cell `data-selected` duplication, real complexity for a cosmetic nuance the row's own non-pinned cells already communicate clearly.
- **Saved Views shipped as a serialization contract only, never a feature** — `DataTableState`/`initialState`/`onTableStateChange` exist and work (confirmed via Playwright capturing a live snapshot), but `DataTable` itself has no name, list, or persistence for a "view," per explicit instruction and consistent with the same boundary already drawn around `Stepper` (no validation) and `AccountMenu` (no owned destinations).
- **Risk-critical and destructive deliberately share the same red hue family** — "critical risk" and "error" are treated as the same severity register, not forced apart for the sake of token independence.
