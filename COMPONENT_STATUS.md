# Krane Design System — Component Status

Generated from a fresh inventory of the current codebase (`app/globals.css`, `components/ui/`, `components/shell/`, `hooks/`, `app/demo/*`, `package.json`, `components.json`) — not recalled from memory, and not a copy of `DESIGN_SYSTEM_STATUS.md`/`DESIGN_SYSTEM_ROADMAP.md`/`FOUNDATION_GAPS.md`, all three of which predate the Typography/Motion/Z-index fixes and the Input/Label builds and are now stale on those points. Documentation only — no code changed to produce this.

## 1. Foundations completed

Flagged across multiple prior audits, not yet acted on. Tracked as a blocker in §4, not listed as completed above.

## 4. Known blockers

- **No `--spacing-*` token scale exists.** Every component's gap/padding is a literal Tailwind utility chosen ad hoc. Doesn't block any component from shipping today, but means there's no scale for a future contributor to check consistency against — the one foundation category repeatedly flagged and never formalized.
- **DataTable virtualization needs a new dependency** (`@tanstack/react-virtual`, not installed) — the first genuinely new npm package on the horizon for this system, only relevant once real data volumes exceed what renders comfortably unvirtualized.
- **No automated test suite and no visual-regression tooling.** Every "verified" claim across this project's history has been build/`tsc`/lint/`curl`/DOM-structure checks plus a manual open-in-browser pass — nothing catches a visual or behavioral regression automatically.
- **`components.json`'s `baseColor: "neutral"`** no longer describes this system accurately (primary is brand yellow, not a neutral-themed shadcn install) — harmless metadata drift today, but would matter if `shadcn add` is ever run again against this config.
- **`RiskIndicator`'s `label` override isn't validated against `level`** — passing contradictory text is possible and silent. Small, accepted gap, not a hard blocker.
- **Mobile/responsive behavior has now been checked and fixed for the App Shell and its composed pieces** (`Sidebar`, `AppHeader`, `OrganizationSwitcher`/`ProjectSwitcher`, `AccountMenu`, `Breadcrumb`, `DataTable`), per `SHELL_RESPONSIVENESS_AUDIT.md`'s findings and the fixes verified against it at 390/768/1440px. This is **not** true of the rest of the system — `Combobox`, `Form`, standalone `Table`, and every other component outside this pass's named scope have never been checked against a small viewport. `Dialog`/`Toast` (inherited shadcn defaults) and `Stepper` (deliberately built) are the only other components confirmed to have real responsive treatment.

## 5. Recommended implementation order

**The entire field-primitive cluster speced in `FORM_SPECIFICATIONS.md` (Input, Label, Textarea, Select, Combobox, RadioGroup, Switch), the `Form` validation layer on top of it, `Alert`, the theme switcher, `Avatar`, `AccountMenu`, `Tabs`, `Breadcrumb`, `Stepper`, DataTable Advanced (Pagination, Row Actions, Empty States, Filters, Sticky Columns), and now Shell responsiveness are all done (§2).** Nothing technical is left blocking any of the work `FORM_FOUNDATION_PLAN.md` originally mapped, plus everything recommended after it. `Button` also picked up its first dedicated demo page this pass — a real, found gap, not a new component. Saved Views' actual feature (naming/persisting/switching views) remains deliberately not built — only its serialization contract exists, per explicit instruction.

1. **Spacing token formalization.** Worth doing once, deliberately, rather than ad hoc — every field primitive above (plus `Form`, `Alert`, the theme switcher, `Avatar`, `AccountMenu`, `Tabs`, `Breadcrumb`, `Stepper`, and DataTable Advanced) shipped without one, so this is now a backfill against sixteen components' worth of un-named spacing values, not a clean-slate exercise.
2. **Card, Progress** — only once a real consuming module surfaces a concrete need for one; this system's own history shows building general chrome speculatively, ahead of an actual requirement, is how scope creep has happened before.
3. **DataTable virtualization, `DataTableColumnHeader` extraction, CSV export, Saved Views (the actual feature), Command Palette, notification-panel content, i18n** — defer as a group until a real requirement forces the question. None of these have a concrete spec today the way everything above does. Mobile/responsive behavior no longer belongs on this list for the App Shell and its composed pieces (fixed per `SHELL_RESPONSIVENESS_AUDIT.md`, see §2) — it remains unaddressed for every other component outside that pass's named scope (`Combobox`, `Form`, standalone `Table`, etc.).

Do not build anything from this list without a fresh, explicit go-ahead — this is a sequencing recommendation, not a standing work order.
