# Krane Design System — Materials Inventory Gap Analysis

Companion to `MATERIALS_INVENTORY_REUSE_REPORT.md`. Per instruction, `DESIGN_SYSTEM_GAP_CONSOLIDATION.md` was re-read in full before starting, and already-documented gaps are not re-investigated below unless this page's build produced genuinely new evidence. Two findings here are new categories of gap; several others are new, concrete evidence for previously-named-but-abstract or previously-open findings; everything else already on record is referenced once, briefly, not re-explained.

---

## New evidence for an already-named-elsewhere gap: risk/status columns sort lexicographically, not by severity

`COMPONENT_STATUS.md` already records, in prose, that `DataTable` "sorts \[status/risk\] columns lexicographically by variant string, not severity rank." None of the three prior page reports tested this directly, because none of them defaulted to or exercised sorting *by* a risk or status column — Procurement Log and Delivery Schedule both used `RiskIndicator`, but never sorted by it.

This page does, and the result is now confirmed with live data, not just described in prose: clicking the "Stockout risk" column header to sort ascending produces `critical, critical, critical, critical, critical, critical, high, high, high, high, …` — alphabetical order (`c` before `h` before `l` before `m`), not severity order (which would put `low` first). Confirmed directly via Playwright reading each row's `data-level` attribute in column order after the sort, not inferred from a screenshot.

**Category: Interaction pattern.** This is a real, concrete defect-in-waiting for any page that lets a user sort by risk or status expecting the most urgent items to surface first — which is precisely what a stockout-risk column exists to support. No workaround was attempted (a custom `sortingFn` could mask this in this page's own column definition, but `DataTable` doesn't expose a way to do this without reaching into TanStack Table's column-definition API directly, and even if it did, fixing it page-by-page would just be the same workaround repeated again — exactly the pattern this whole project's instructions ask to avoid). Left as a documented gap.

## New category of gap: no progress/gauge visualization

**Category: Component, and Data Visualization** (the latter previously named in `DESIGN_SYSTEM_GAP_CONSOLIDATION.md` as "an open question, not a confirmed gap" — this page resolves that question with a concrete case). Confirmed by the same full-inventory method used for every prior component audit: no `Progress`, gauge, or any other proportional/quantity-visualization primitive exists in `components/ui/`. `COMPONENT_STATUS.md` already lists `Progress` alongside `Card` as "no spec, no code," but — like `Card` before Procurement Log's KPI-strip need — it had never had a concrete, page-level use case driving it. This page is that use case: "quantity on hand relative to reorder point" is closer to a textbook `Progress`-bar scenario than anything in the prior three pages came near. This page shows the relationship as two plain numbers (Qty on hand, Reorder point columns) plus the derived `RiskIndicator` instead — a reasonable substitute built entirely from existing components, not an equivalent one. No bar, gauge, or other visual proportion indicator was hand-rolled from raw `div`s to fill the gap; per instruction, the absence is documented instead.

## New, concrete evidence for an already-named gap: numeric-range filtering

`DESIGN_SYSTEM_GAP_CONSOLIDATION.md` already names "no numeric-range filter variant" as P3 — real but, as of Delivery Schedule, "not yet a recurring cost." This page is the first where the absence is squarely central rather than peripheral: the two most natural filters for an inventory page — "show me materials at or below their reorder point" and "show me materials over $X in unit cost" — both require comparing a row's own two numeric fields, or a numeric field against a threshold, neither of which `DataTable`'s three filter variants (`text`/`select`/`dateRange`) can express. Confirmed by re-reading `DataTableFilterVariant`'s type union in `components/ui/data-table.tsx`, unchanged since Procurement Log's original finding. **Category: Variant.** This page does not filter by quantity or cost as a result — the four filters it does have (Category, Status, Warehouse, Vendor) are all `select`, the same workaround-via-bounded-enumeration every prior page used.

## New finding, carefully distinguished from an existing component of the same general shape: no numeric increment/stepper control

**Category: Interaction pattern.** A real inventory page wants a quick "+/− quantity" control for the common "bump stock up or down by a small amount" interaction. No such control exists in this design system under any name. This is **not** the same gap as anything related to `components/ui/stepper.tsx`'s existing `Stepper` — that component is a multi-step wizard/progress-tracker indicator (Draft → In Review → Done), an entirely different concept that happens to share a common name with a numeric-increment control in some other design systems, but not in this one. Naming this distinction explicitly because the shared name is a real, findable source of future confusion: a future engineer searching this codebase for "stepper" to solve a quantity-adjustment problem will find `Stepper` and could reasonably assume it's the wrong tool for the wrong reason, rather than confirming no tool exists at all. This page's "Adjust stock" row action remains a toast, identical to how "Edit" has worked on every prior page — no `+`/`-` buttons were built to fill the gap.

## New, precise refinement of an already-named gap: `Badge`'s real ceiling is "four-or-five semantic meanings plus one neutral slot," not strictly five

Already covered in full in the Reuse Report. **Category: Variant.** Restated here only as the gap-analysis framing: the existing finding ("Badge has exactly five non-default variants, a real ceiling") slightly overstates how tight the ceiling is in practice, since `default` is always available as a genuinely distinct, non-semantic sixth slot — this page is the first to actually use it for that purpose. The practical ceiling for *semantic* (health/urgency) meanings is four-or-five; a status that isn't really a health signal (here, "Discontinued" — inactive, not unhealthy) has a real home in `default` without needing a new token or variant at all. Worth writing down precisely so the next page author doesn't treat the original "five" finding as tighter than it actually is.

## Carried forward, reconfirmed, not re-explained

Each of the following was found on an earlier page, reconfirmed identically on this one, and is documented in full in `DESIGN_SYSTEM_GAP_CONSOLIDATION.md` — listed here only to record that this page's build reproduced the same need, not to repeat the reasoning:

- **Component**: no `Card`/stat-tile, no Drawer/Sheet, no exported `DataTableColumnHeader`, no permission/role-gating primitive.
- **State**: no in-flight visual state on rows during a bulk action (`Reorder`/`Discontinue` on this page, same as every prior page's bulk actions).
- **Token**: no `--spacing-*` scale (this page's every `gap-2`/`p-4`/`px-3` is exactly as ad hoc as the prior three).
- **Interaction pattern**: no row-click-to-view, no inline cell editing, no column resize/reorder, no confirm-before-bulk-action step, no page-level action overflow pattern (this page's three actions again needed `flex-wrap`, exactly as Procurement Log's did).
- **Accessibility requirement**: `DataTableRowActionsMenu`'s static `aria-label="Row actions"` (reconfirmed a fourth time, independent of this page's specific column layout); `SidebarItem`'s missing `aria-current="page"` (reconfirmed a fourth time — "Inventory" was marked active the same styling-only way as every prior page's active item); no skip-to-content link in `AppShell`; `RiskIndicator`'s `label` not validated against `level` (relevant again, since this page uses `RiskIndicator` on every row); `Button`'s missing `loading` prop (reconfirmed a fourth time on the "Add material" submit button); `DialogContent`'s missing default `overflow-y-auto` (reconfirmed a fourth time, same manual `className` override needed).
- **Confirmed working, not a gap, again**: `AppShell`/`Sidebar`/`AppHeader` responsive behavior and `DataTable`'s contained-overflow, at 390/768/1440px in both light and dark mode; the `Combobox`-inside-`FormField` `aria-labelledby` integration; and — newly relevant after the prior fix pass — the `DataTable` sticky-column defect did **not** reproduce on this page (confirmed via the same computed-style check: exactly two pinned headers, one per side), the first page built since that defect was resolved.

## No defects fixed, no foundations modified

Consistent with instruction: nothing in `components/ui/`, `components/shell/`, `app/globals.css`, or any other design-system foundation file was touched to build this page. Every finding above was worked around using an existing primitive in a documented, non-ideal way, or left unaddressed and written down here instead.
