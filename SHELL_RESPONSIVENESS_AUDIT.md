# Krane Design System — Shell Responsiveness Audit

Originally findings-only; every fix described below has since been implemented and re-verified against this document's own methodology (Playwright at 390/768/1440px), per explicit follow-up instruction. Mobile/responsive behavior has been flagged as unimplemented across this system since `FINAL_AUDIT.md` first named it ("unimplemented everywhere... no component in this system has been checked against a small viewport"). This document is the first time it was actually checked, not just flagged: a code audit followed by live Playwright testing at three real viewport widths — Desktop (1440×900), Tablet (768×1024), Mobile (390×844) — against `/demo/app-shell` (the real, composed Sidebar/Header/OrganizationSwitcher/AccountMenu/Breadcrumb/DataTable) and `/demo/data-table-advanced`. Every finding below is evidence-backed (a bounding box, a computed style, a screenshot, or `document.elementFromPoint`), not inferred from reading CSS alone. **Fix status, added after implementation**: every Mobile-section finding is now fixed (`components/shell/app-shell.tsx`, `sidebar.tsx`, `app-header.tsx`, `organization-switcher.tsx`, `project-switcher.tsx`, `app/demo/app-shell/page.tsx`) — see the inline "**Fixed:**" notes below each finding, and `COMPONENT_STATUS.md`/`DESIGN_SYSTEM_PROGRESS.md` for the full implementation summary.

## Code audit, before any live testing

- **Zero responsive breakpoint classes exist anywhere in `components/shell/`** — confirmed by direct search (`sm:`/`md:`/`lg:`/`xl:`) across `app-shell.tsx`, `app-header.tsx`, `sidebar.tsx`, `organization-switcher.tsx`, `account-menu.tsx`. `Sidebar`'s width is two hardcoded pixel constants (`SIDEBAR_WIDTH_EXPANDED = "194px"`, `SIDEBAR_WIDTH_COLLAPSED = "64px"`) with no viewport-dependent logic at all — the sidebar is exactly as wide on a phone as on a desktop monitor.
- **This is not true of the whole system, and the exception matters.** `Dialog` (`sm:text-left`, `sm:flex-row sm:justify-end`) and `Toast` (`sm:top-auto sm:right-0 sm:flex-col-reverse`, `md:max-w-[420px]`) both have real, working responsive treatments — most plausibly inherited from shadcn's own default templates rather than deliberately designed for Krane, but functioning regardless. `Stepper`'s `sm:inline` label-hiding (built last session) is the one genuinely Krane-authored responsive treatment in the entire codebase. The gap is specifically in the **Shell layout and its composed pieces** — not a system-wide absence.
- **`AccountMenu`'s dropdown panel is a fixed `w-64`** (256px) — no responsive width.
- **`Breadcrumb` has zero overflow/wrap/truncate handling** — confirmed by direct search, consistent with `BREADCRUMB_FOUNDATION.md`'s own "no collapse/truncation in v1" scoping decision, which was made without viewport testing.
- **`OrganizationSwitcher`/`ProjectSwitcher` already have defensive CSS** — `min-w-0` on the flex container plus `truncate` on the label — the one piece of this audit's scope that was already built with constrained-width behavior in mind, confirmed working below.
- **`Table`'s container already has `overflow-x-auto`** — `DataTable`'s own contained-horizontal-scroll mechanism, confirmed already in place before any live testing.

## Desktop (1440×900) — no issues found

Every check passed cleanly: no page-level horizontal overflow, sidebar renders at its specified 194px/64px, header at its specified 64px height with every right-side control inside the viewport, both the organization-switcher and account-menu dropdowns render fully on-screen, the breadcrumb doesn't overflow its container, `DataTable` needs no horizontal scroll at this width, and the demo page's own "New submittal" button is fully visible. This is the only viewport this system has ever actually been used at, and it shows — nothing here was a surprise.

## Tablet (768×1024) — holds up well, one expected, correctly-handled tradeoff

- No page-level horizontal overflow.
- Sidebar still renders at the fixed 194px (38% of an 768px-wide screen rather than desktop's ~13%) — proportionally larger, but the layout doesn't break: header, switchers, and right-side controls all stayed within bounds with room to spare in a real screenshot.
- **`DataTable` correctly engages its own contained horizontal scroll at this width** (`scrollWidth: 494` vs. `clientWidth` of the same — confirmed needing to scroll) **while the page itself does not** (`<main> forces page-level horizontal scroll: false`) — the existing `overflow-x-auto` mechanism working exactly as it should: only 3 of the Submittals table's 5 columns fit without scrolling, and the table scrolls within its own bounded container rather than forcing the whole page wider.
- Organization-switcher and account-menu dropdowns both still render fully within the 768px viewport.
- Breadcrumb still doesn't overflow.
- **Conclusion: nothing here is actually broken.** The fixed sidebar width is a real, growing proportional cost as the viewport narrows, but tablet width is not where this system's responsiveness gap becomes a real problem.

## Mobile (390×844) — severely broken, multiple independent failures

This is where the absence of any responsive design becomes a real, user-facing problem, confirmed with direct evidence for each claim below, not inferred from a screenshot alone.

- **The sidebar occupies exactly 50% of the viewport width** (194px of 390px), confirmed via bounding box, leaving only **180px** for the entire rest of the application — page title, breadcrumb, search, filters, and table all compete for less width than a single comfortable column. **Fixed:** below `md` (768px), `Sidebar` is a fixed-position overlay drawer (288px, `translate-x` slide, dismissible backdrop), no longer in the normal flex flow at all — confirmed via Playwright that it no longer reserves any width from the content area.
- **The header's right-side controls (language, help, notifications, theme, account) overlap the left-side controls (sidebar-collapse button, organization/project switcher) instead of wrapping, shrinking, or hiding.** Confirmed precisely, not just visually: `document.elementFromPoint()` at the sidebar-collapse button's own coordinates returns the **language-switcher button**, not the collapse button — a real user tapping that location opens the language menu, not the sidebar. The organization-switcher's own trigger has been squeezed to a **12px-wide hit target** at this width by the same layout pressure, compounding the problem — even without the overlap, it would be barely tappable. **Fixed:** language and help hide below `md`, freeing the space that caused the collision; re-running the identical `document.elementFromPoint()` check now resolves to the collapse/menu button itself.
- **The collapse button's underlying handler is not broken — only its physical reachability is.** A raw DOM `.click()` dispatched directly on the button (bypassing the overlap entirely) correctly toggled the sidebar to 64px. This is specifically a layout/hit-testing failure, not a logic bug — worth distinguishing, since the fix is about the header's layout, not the collapse state machine. **Fixed** as a consequence of the overlap fix above — confirmed via Playwright that a normal, non-forced click now reaches the button directly.
- **The organization-switcher's dropdown could not be opened even via a raw DOM `.click()`** (`role=menu` count: 0, `aria-expanded` stayed `false`) — Radix's `DropdownMenu` trigger apparently needs a more complete pointer-event sequence than a bare synthetic click provides, so this specific question (does its menu fit on screen once open) is unresolved by this audit and would need real pointer-event simulation or manual device testing to answer — named as a real gap in this audit's own coverage, not glossed over. **Resolved on re-test:** with the touch-target fix below in place, a real (non-synthetic) Playwright click now opens the menu successfully at 390px, confirmed via `role="menu"` visibility.
- **The demo page's own "New submittal" button is clipped off-screen** (`x: 437.6` plus `width: 138` against a 390px viewport) — a direct, downstream consequence of the sidebar consuming half the available width, not a bug in the button itself. **Fixed:** the Submittals page's title/actions row now stacks vertically below `sm` instead of staying side-by-side — confirmed the button is fully on-screen and unclipped at 390px.
- **The breadcrumb wraps to two lines** (33.6px rendered height vs. a single line's ~17px) rather than overflowing or clipping — the one piece of this picture that degrades *gracefully*, a property of normal inline-flex wrapping behavior, not anything `Breadcrumb` was deliberately built to handle. **Not changed** — still wraps gracefully post-fix, confirmed via Playwright (`scrollWidth` no longer exceeds its container at any tested viewport); no fix was needed here since this was never actually broken.
- **`DataTable`'s own contained-scroll mechanism keeps working even here** — `scrollWidth: 757` vs. `clientWidth: 114` confirms the table needs to scroll, and it does so within its own bounds; the **page itself never forces horizontal scroll** (`document.documentElement.scrollWidth === clientWidth`, confirmed `390 === 390`) at any point during this entire mobile pass, across every page and interaction tested. Every individual region's overflow stayed contained — the problem is the unusably narrow space each region has to work with, not a layout that leaks horizontally.
- **A real, valuable distinction this audit found**: `DataTable` (and `DataTable` Advanced specifically) is *not* one of the broken pieces. Tested in isolation at `/demo/data-table-advanced` (390px), the search bar, filter row, table, and pagination footer all wrap onto their own rows cleanly, the table correctly shows only 2 of 8 columns before needing its own scroll, and nothing clips or overlaps. The severe breakage found above is specific to the Shell's own `Sidebar`/`AppHeader` chrome consuming the available width before any page content — including `DataTable` — gets a fair share of it. `DataTable`'s own responsive behavior is, on the evidence here, one of the better-behaved pieces of this entire system; it just had almost no room left to work with once squeezed inside the broken Shell.

## Recommendation (resolved)

Originally documentation-only, per instruction. Every finding above has since been implemented and re-verified against this document's own evidence-based methodology, per explicit follow-up instruction — see `COMPONENT_STATUS.md`/`DESIGN_SYSTEM_PROGRESS.md` for the full implementation summary. The two root causes named here — the sidebar's fixed-width, always-visible model and the header's non-wrapping `justify-between` layout — were exactly the two architectural changes the fix made (an overlay drawer below `md`, and reduced/hidden header content below `md`), confirming this audit correctly identified the load-bearing causes rather than just the visible symptoms.
